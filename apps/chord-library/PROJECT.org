* Overview
Six standalone HTML offline copies of musicalwonders.org apps. No build
step, no bundler — edited in place, viewed directly in the browser.
=pitch-spelling.js= is a shared module pulled in via plain =<script>= tag.

Main files:
- =chord-sheet.html= — Chord Sheet Builder (sidebar + preview)
- =director.html=    — Director app (drones, key, solfege, etc.)
- =pitch-spelling.js= — Shared LOF-based spelling module
- =index.html=       — Landing
- Two others — see =MEMORY.md=

* Chord Sheet Builder                                              :chord-sheet:
** DONE Enharmonic preservation in chord input
Stored =rootName= alongside =rootPc= in parsed tokens so "Bbm A#m"
renders both spellings instead of normalizing.
=parseChordToken= → =buildDiagramCard= path.

** DONE Constant-height keyboard diagrams
Flex layout + inline px SVG dims; print CSS uses =!important= to
revert to fill-column. =renderKeyboard= sets =style.width= / =style.height=.

** DONE Voice leading
Cross-chord economy of motion via inv × oct search per keyboard chord and
per-position score for fretted. Sum-of-nearest-neighbor distance.
- Toggle: "Favor voice leading"
- Triggers on parse, append, remove, instrument enable, tuning change,
  style change, toggle change.

** DONE Voicing styles
- Standard / Shell (drops 5 for 7th chords) for all instruments
- Triad (high / mid / low / bass) string-set filter for fretted
- Per-instrument select, with its own position-cache key

** DONE Fingerings — fretboard
Algorithmic: barre = 1, then remaining fretted notes get 2/3/4 in
ascending fret order. Rendered below string labels at y=432.

** DONE Fingerings — keyboard
RH / LH / two-hand modes. Two-hand colors LH red so it reads at a glance.
- Presets RH bottom→top: [1], [1,5], [1,3,5], [1,2,3,5], [1,2,3,4,5]
- Presets LH bottom→top: [5], [5,1], [5,3,1], [5,3,2,1], [5,4,3,2,1]
- Two-hand split puts LH on bass, RH on upper structure.

** DONE Roman numeral chord entry
Sidebar input; parses tokens like "V7 ii bVII viio7 #ivo VM7" against
=currentKeyContext()=.

** DONE Key panel (auto-detect / manual)
Krumhansl-Schmuckler major-profile correlation on a pc-class histogram
built from current chords. Manual override via select.

** DONE Solfege labels
Moveable-do chromatic (Do Ra Re Me Mi Fa Se Sol Le La Te Ti); swapped
into every diagram label via =labelForPc(pc, fallback)=.

** TODO Lookup table for common fretboard fingerings
Algorithmic picks 3-3-4 where a player would use 2-3-4 in some stretches.
Small override table keyed by canonical shape would fix the outliers
without giving up the algorithm for unusual voicings.

** TODO Better "no voicing found" UX for strict triad filters
=triad-bass= on a 7th chord can return empty positions. Currently shows
"no voicing found" — could fall back to next-widest set with a note,
or grey out unavailable styles per chord.

* Student Tracking System                                  :student-tracking:
Big idea: track per-student exposure across four parallel inventories so
chord sheets can be differentiated by novelty level. End state: Supabase
backend + Quarto site.

** Inventories
1. Chord types (e.g., maj7, m7b5)
2. Voicings (instrument-specific shapes/inversions)
3. Comping patterns
4. Melodic improvisation patterns

** TODO Schema design                                               :backend:
Recommended shape: four catalogs + students + four parallel exposure
tables + songs + sessions. Polymorphic exposures table is tempting but
loses FK integrity; partitioned with a union view keeps RLS clean.

Sketch:
#+begin_src sql
  students(id, name, primary_instrument, started_at, notes)

  catalog_chord_types(id, label, symbol, intervals, formula, family)
  catalog_voicings(id, instrument, tuning, chord_type_id FK,
                   style, shape_key, difficulty, label)
  catalog_comping_patterns(id, label, genre, time_sig, rhythm_json, difficulty)
  catalog_improv_patterns(id, label, scale, mode, contour, range_st, difficulty)

  exposures_chord_types(student_id, type_id, count, first_seen, last_seen,
                        last_song_id, notes,
                        PRIMARY KEY(student_id, type_id))
  exposures_voicings(...)
  exposures_comping(...)
  exposures_improv(...)

  songs(id, title, key_tonic, key_mode, tempo, source, notes)
  song_items(song_id, category, item_id)
  sessions(id, student_id, song_id, date, level_target, teacher_notes)
#+end_src

** TODO Differentiation algorithm                                  :ui:
#+begin_src text
  for a song S and student X with novelty target N:
    required  = song_items(S)
    seen      = union of student's exposures across categories
    novel_set = required - seen
    if |novel_set| < N:
      substitute K items with adjacent-difficulty unseen ones,
      subject to musical validity (same chord, different voicing)
#+end_src
Lives in the app, not the DB. Needs catalog + exposures read.

* Workflow Notes
- Chord-sheet, director, and pitch-spelling.js share LOF + key context conventions.
- Position cache keys are =tuning.id + ':' + styleId=; clear cache when style changes.
- Manual voicing-dropdown changes persist until next chord/instrument/tuning/style change.
- LH fingerings render in red (=#c0392b= on white keys, =#ff8c8c= on black) so two-hand mode is glance-readable.
